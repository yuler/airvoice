# 背景

## 问题

桌面设备通常缺少可靠的麦克风：台式机没有麦克风、蓝牙耳机不稳定，或者你想边走边口述而眼睛盯着显示器。手机已经是更好的麦克风，具有更强的降噪能力。差距在于如何将识别到的**文字**输入到 Mac 或 Linux 上获得焦点的应用中。

## 灵感

此项目始于一次 [Gemini 对话](https://gemini.google.com/share/62257304d6bd)，探索远程语音输入：手机作为麦克风，PC 作为接收器，解决将麦克风连接到桌面很麻烦的场景。

## 产品理念

**Airvoice 是桥接工具，不是语音引擎。**

- **移动端：** 带文本框的 iOS 应用；用户通过第三方语音键盘（**豆包输入法**为主，**微信输入法**备选）口述。
- **桌面端：** macOS / Linux 上的 Go CLI；通过局域网 WebSocket 接收文本并粘贴到光标位置。
- **MVP 不做自定义语音识别** — 利用现有输入法的质量，而不是训练或托管模型。

## 为什么不使用现有工具？

| 方案 | 示例 | 局限性 |
|------|------|--------|
| 虚拟麦克风 | WO Mic、AudioRelay | 内核驱动/音频管道；不是文本桥接 |
| 跨屏输入法 | 讯飞输入法 | 仅云端、输入法锁定、Linux 支持差 |
| 远程控制 | Unified Remote | 以远程控制为主，不专注于打字桥接 |

**Airvoice 差异化：** 仅局域网、Linux 友好、无自定义语音识别、使用你已信任的键盘。

## 设计决策（已锁定）

| 主题 | 决策 |
|------|------|
| 产品名称 | **Airvoice** |
| 移动端 | iOS SwiftUI, iOS 17+ |
| 桌面端 | Go CLI (`airvoice serve`) |
| 配对 | QR 码 + 一次性 token（MVP 不做 mDNS） |
| 发送模式 | 键盘消失或文本空闲 1.5 秒时自动发送 |
| 多行 | 原样粘贴，保留 `\n` |
| Linux | MVP 支持 X11 + Wayland |
| 语音识别 | 仅第三方输入法 |
